# Концепция Windows-приложения n8nRestore

## Цель

Создать Windows-приложение, которое проводит пользователя через полное восстановление n8n/Selfhost AI на новый VPS, показывает прогресс, запрашивает подтверждения и формирует итоговый отчёт.

## Рекомендуемая архитектура

Windows-приложение выступает SSH/SFTP-оркестратором, а recovery engine выполняется локально на VPS.

Приложение:

- проверяет SSH host fingerprint и параметры VPS;
- загружает проверенную версию recovery engine;
- запускает восстановление как временный systemd unit;
- показывает snapshots, этапы и предупреждения;
- отображает Tailscale login URL;
- принимает выбор домена и показывает hostname mapping;
- передаёт подтверждения `ACTIVATE` и Telegram;
- проверяет DNS, HTTPS, контейнеры, SFTP и новый Restic snapshot;
- подключается повторно к уже выполняющемуся восстановлению после обрыва SSH.

Recovery engine:

- выполняет Linux-операции, retries, rollback/cleanup и проверки;
- хранит состояние на VPS;
- выдаёт машиночитаемые JSON events;
- не принимает секреты через аргументы командной строки.

Пример событий:

```json
{"stage":"tailscale","status":"waiting_for_login","url":"https://login.tailscale.com/..."}
{"stage":"snapshot","status":"waiting_for_selection","snapshots":[]}
{"stage":"postgres","status":"completed"}
{"stage":"activation","status":"waiting_for_confirmation"}
```

## Почему не передавать весь набор команд напрямую

Прямые короткие SSH-команды подходят для bootstrap, диагностики, запуска процесса и чтения статуса. Полное восстановление включает длительные операции, traps, retries, временные секреты, cleanup и возобновление после разрыва связи. Эту транзакционную логику должен выполнять процесс на VPS, независимый от SSH-сессии.

Если Bash в будущем станет ограничением, recovery engine можно заменить отдельным статическим Linux-агентом. Протокол состояний Windows-приложения при этом должен сохраниться.

## Безопасность

- Пароли Restic/SFTP хранить только в памяти, если пользователь явно не выбрал защищённое сохранение.
- Для сохранения на Windows использовать DPAPI с привязкой к текущему пользователю.
- Не помещать пароли и токены в command line, JSON events или журналы.
- Временные серверные файлы создавать с правами `600` и удалять при завершении.
- Не принимать SSH host key автоматически.
- Отдельно подтверждать запуск production и перенаправление внешних webhook.

## Предлагаемый стек

- WPF на актуальном LTS .NET для Windows UI;
- SSH.NET для SSH/SFTP;
- self-contained Windows build;
- systemd transient unit для независимого выполнения recovery на VPS;
- JSON Lines для событий и команд управления.

## Первый этап разработки

Сначала формализовать state machine recovery: этапы, входные параметры, события, подтверждения, повторное подключение и терминальные состояния. После стабилизации протокола создавать UI.
