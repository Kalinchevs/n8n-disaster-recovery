# Инструкция по восстановлению n8n из резервной копии

Эта инструкция предназначена для аварийного или тестового восстановления текущей установки n8n на новый VPS с помощью `restore-n8n.sh`. Сохраните ссылку на неё отдельно от основного сервера.

Репозиторий: <https://github.com/Kalinchevs/n8n-disaster-recovery>

## 1. Что восстанавливает скрипт

Комплект скриптов отслеживает фактический набор включённых стандартных профилей Selfhost AI по значению `COMPOSE_PROFILES`. Состав резервной копии формируется с учётом реальной конфигурации: данные включённых сервисов добавляются в backup, а отключённые необязательные сервисы пропускаются. При восстановлении welcome page создаётся заново по тому же сохранённому набору профилей, поэтому на ней должны отображаться только реально используемые сервисы.

Скрипт восстановления разворачивает систему из зашифрованного репозитория Restic на Synology:

- проект Selfhost AI и его `.env`;
- PostgreSQL, включая базы, роли и данные n8n;
- постоянный том n8n и прежний `N8N_ENCRYPTION_KEY`;
- Portainer и тома Caddy;
- Playwright MCP;
- данные включённых стандартных профилей Selfhost AI;
- SSH-файлы для доступа к Synology;
- ежедневный backup-скрипт, systemd service и timer;
- welcome page, соответствующую сохранённому `COMPOSE_PROFILES`;
- все первоначально опубликованные workflow восстанавливаются как `Unpublished`;
- обычные workflow публикуются только после отдельного подтверждения `WORKFLOWS`;
- workflow с Telegram Trigger остаются изолированными до отдельного подтверждения `TELEGRAM`;
- Telegram webhook переносится только после этого подтверждения.

Redis и Docker-образы не восстанавливаются из копии: Redis создаётся заново, а образы загружаются из registry.

## 2. Важные предупреждения

1. Запускайте скрипт только на новом или специально подготовленном VPS с Ubuntu 24.04 LTS amd64.
2. Не устанавливайте Selfhost AI или n8n заранее. Каталог `/root/selfhost-ai` должен отсутствовать.
3. Не допускайте одновременной работы старого и восстановленного n8n с одинаковыми опубликованными workflow. После `ACTIVATE` все восстановленные workflow остаются `Unpublished`, но подтверждение `WORKFLOWS` может запустить расписания, webhooks и другие внешние действия.
4. Не вводите `ACTIVATE`, пока не решили, какая машина должна стать активной.
5. Не вводите `TELEGRAM`, пока старый VPS не остановлен и DNS не направлен на восстановленный VPS.
6. После `ACTIVATE` скрипт автоматически включает backup-таймер, если восстановленный SSH-ключ обеспечивает доступ к Synology. При тестовом восстановлении это может создать второй набор резервных копий в том же Restic-репозитории.
7. Не удаляйте старый VPS до завершения всех проверок и создания новой резервной копии на восстановленной машине.

## 3. Что подготовить заранее

Понадобятся:

- новый VPS: Ubuntu 24.04 LTS, архитектура amd64, root-доступ;
- доступ VPS в интернет;
- доступ к аккаунту Tailscale, в котором находится Synology;
- пароль репозитория Restic, сохранённый отдельно от VPS;
- пароль пользователя SFTP на Synology;
- возможность проверить SSH fingerprint Synology;
- доступ к управлению DNS прежнего или выбранного нового публичного домена;
- при использовании закрытого пользовательского образа n8n — GitHub token с правом `read:packages` для входа в `ghcr.io`;
- публичный IPv4 нового VPS;
- доступ к старому VPS или возможность гарантированно его выключить.

Значения, используемые в текущей схеме по умолчанию:

| Параметр | Значение |
|---|---|
| Synology Tailscale/MagicDNS | `homenas` |
| SFTP-пользователь | `n8n_backup` |
| Путь Restic | `/n8n-backups/restic-n8n` |
| Текущий домен n8n | `n8n.codecraftsergo.com` |
| Каталог Selfhost AI | `/root/selfhost-ai` |
| Каталог журнала восстановления | `/var/log/n8n-disaster-recovery` |

Пароли, токены, приватные SSH-ключи и `N8N_ENCRYPTION_KEY` в GitHub не хранятся.

## 4. Загрузка и запуск скрипта

Подключитесь к новому VPS по SSH от имени `root`. Не создавайте `/root/selfhost-ai` и не запускайте установщик Selfhost AI вручную.

Запустите скрипт:

```bash
curl -fsSL https://raw.githubusercontent.com/Kalinchevs/n8n-disaster-recovery/main/restore-n8n.sh -o /tmp/restore-n8n.sh && bash /tmp/restore-n8n.sh
```

Скрипт создаёт подробный журнал вида:

```text
/var/log/n8n-disaster-recovery/recovery-YYYYMMDD-HHMMSS.log
```

При ошибке сохраните путь к журналу и последние строки сообщения.

## 5. Прохождение интерактивного мастера

### 5.1. Начальное подтверждение

Скрипт выводит назначение, версию и путь к журналу. На вопрос `Continue on this VPS?` подтвердите продолжение только после проверки машины.

После этого скрипт:

- проверит root-доступ, ОС и архитектуру;
- установит базовые пакеты;
- установит Docker Engine и Docker Compose;
- установит Tailscale.

### 5.2. Подключение к Tailscale

Если VPS ещё не авторизован, скрипт запустит Tailscale и покажет ссылку для входа. Откройте её в браузере, авторизуйте новую машину и вернитесь в терминал.

Одна попытка авторизации длится не более пяти минут. Если control plane Tailscale временно недоступен, ссылка устарела, устройство ожидает административного одобрения или регистрация не завершилась по другой причине, мастер предложит повторить попытку. При повторе используется `tailscale up --force-reauth`, поэтому открывайте только новую показанную ссылку.

Успешная проверка выглядит примерно так:

```text
pong from homenas (100.x.x.x) ...
OK: Synology is reachable over Tailscale.
```

После восстановления отключите истечение ключа для этой машины в Tailscale Machines, если VPS должен работать постоянно.

### 5.3. Параметры Synology и Restic

Мастер запросит:

1. `Synology Tailscale IP or MagicDNS name` — обычно можно оставить `homenas`.
2. `Synology SFTP user` — обычно `n8n_backup`.
3. `Restic repository path on Synology` — обычно `/n8n-backups/restic-n8n`.
4. Пароль Restic.
5. Пароль SFTP-пользователя Synology.

Ввод паролей в терминале не отображается — это нормально. Пароль Restic сохраняется с правами `600`; временный SFTP-пароль хранится только в `/run` и удаляется при завершении скрипта.

### 5.4. Проверка SSH fingerprint

Скрипт покажет fingerprints SSH-ключей Synology. Сравните их с известными значениями или с ключами, показанными на доверенной машине.

Для продолжения необходимо точно ввести:

```text
YES
```

Не подтверждайте неизвестный fingerprint.

### 5.5. Выбор снимка Restic

Скрипт показывает 10 последних снимков с датой и временем по Москве, логическим размером, тегом и коротким ID.

Варианты выбора:

- `Enter` или `latest` — самый новый снимок;
- число `1`–`10` — строка из таблицы;
- дата `YYYY-MM-DD` — самый новый снимок за указанную дату, в том числе среди более старых копий.

Если восстанавливается аварийно потерянный сервер, обычно выбирайте самый новый успешный снимок. Если причина аварии могла попасть в последний бэкап, выберите заведомо исправную дату.

На вопрос о проверке целостности репозитория рекомендуется ответить `yes`. Проверка увеличивает длительность восстановления, но выявляет повреждения до записи данных на новый сервер.

### 5.6. Выбор публичного домена

После извлечения Selfhost AI мастер показывает прежний базовый домен из `N8N_HOSTNAME`.

- Нажмите Enter, чтобы сохранить все восстановленные hostname без изменений.
- Введите другой базовый домен без `https://`, пути и порта, чтобы перенести публичные сервисы, например `example.com`.

При смене домена скрипт покажет рассчитанные изменения всех заполненных `*_HOSTNAME`, которые используют прежний доменный суффикс. Префиксы сервисов сохраняются. Например:

```text
N8N_HOSTNAME                 n8n.old.example -> n8n.new.example
PORTAINER_HOSTNAME           portainer.old.example -> portainer.new.example
```

Проверьте список и точно введите:

```text
DOMAIN
```

Если подтверждение не введено, `.env` остаётся с прежними hostname. Значения с другим доменным суффиксом, пустые и нестандартные значения мастер не меняет. После изменения скрипт валидирует Docker Compose и проверяет, что рассчитанный `WEBHOOK_URL` совпадает с `https://<N8N_HOSTNAME>`.

До `ACTIVATE` создайте или подготовьте DNS-записи для показанных hostname. Caddy сможет получить новые сертификаты только при корректных DNS-записях и доступных извне портах 80/443.

### 5.7. Автоматическая часть восстановления

Далее вмешательство обычно не требуется. Скрипт:

- скачивает выбранный снимок;
- проверяет наличие и целостность дампа PostgreSQL;
- восстанавливает Selfhost AI и `.env`;
- проверяет наличие `N8N_ENCRYPTION_KEY`;
- применяет подтверждённый базовый домен и проверяет итоговый `WEBHOOK_URL`;
- перегенерирует welcome page;
- восстанавливает registry credentials, если они были в снимке;
- запускает PostgreSQL и Redis;
- восстанавливает базы;
- при остановленном n8n сохраняет список всех первоначально опубликованных workflow, устанавливает им `active=false` и удаляет их восстановленные маршруты из `webhook_entity`;
- восстанавливает том n8n, Portainer, Caddy и данные активных профилей;
- восстанавливает Playwright и подключает его к фактической Docker-сети n8n;
- адаптирует backup-скрипт к найденным Docker volumes;
- восстанавливает systemd service и timer;
- проверяет SFTP-доступ по восстановленному SSH-ключу.

Во время импорта PostgreSQL терминал показывает одну обновляемую строку с прошедшим временем. Полный вывод `psql`, включая предупреждения и список восстановленных баз, записывается в:

```text
/srv/n8n-recovery/YYYYMMDD-HHMMSS/postgres-restore.log
```

Если импорт или последующая проверка баз завершится ошибкой, recovery-скрипт остановится и выведет точный путь к этому файлу.

При скачивании и сборке Docker-образов Compose использует тихий формат прогресса. Строки состояния отдельных слоёв (`Pulling`, `Downloading`, `Extracting`) скрываются, но предупреждения и ошибки остаются в терминале и общем recovery-журнале.

Если закрытый образ n8n невозможно скачать, войдите в GHCR только на новом VPS:

```bash
docker login ghcr.io -u Kalinchevs
```

В качестве пароля используйте GitHub token с минимальным правом `read:packages`. Не передавайте токен в аргументах команд и не публикуйте его в журнале.

## 6. Точка `ACTIVATE`

Перед запуском рабочих сервисов скрипт останавливается и показывает сводку, включая число восстановленных как `Unpublished` workflow. До ввода `ACTIVATE` основной n8n, workers, task runner, Playwright и backup-таймер не запущены.

### Аварийное переключение

Перед активацией:

1. Остановите старый VPS либо убедитесь, что он недоступен.
2. Убедитесь, что два экземпляра n8n не смогут выполнять одни и те же workflows.
3. Подготовьте изменение A-записи домена на публичный IP нового VPS.

Для запуска восстановленной системы точно введите:

```text
ACTIVATE
```

После этого скрипт запускает Selfhost AI, Playwright и, при исправном SFTP-доступе, ежедневный backup-таймер. Все восстановленные workflow остаются `Unpublished`, поэтому одно подтверждение `ACTIVATE` не запускает их и не меняет webhook production-бота.

Затем скрипт показывает первоначально опубликованные workflow без Telegram Trigger. Чтобы вернуть им состояние `Published`, точно введите:

```text
WORKFLOWS
```

Нажатие Enter оставляет их `Unpublished`. Telegram-workflow в эту операцию не входят и требуют отдельного подтверждения.

### Тестовое восстановление

Если старая боевая машина продолжает работать, можно ввести `ACTIVATE`, но не вводить `WORKFLOWS` и `TELEGRAM`. В этом режиме интерфейс и сервисы доступны для проверки, а восстановленные workflow остаются `Unpublished` и не должны выполнять внешние действия.

Сразу после запуска тестовой машины учитывайте автоматический backup-таймер. Если тестовая машина не должна создавать копии в боевом репозитории, отключите его:

```bash
systemctl disable --now n8n-daily-backup.timer
```

Не вводите `WORKFLOWS` или `TELEGRAM` и не публикуйте workflow вручную при безопасном тесте. Запуск тестового n8n с восстановленными workflow в состоянии `Unpublished` не должен затрагивать production-триггеры и Telegram webhook.

## 7. DNS, HTTPS и Telegram webhook

После `ACTIVATE` направьте A-запись выбранного `N8N_HOSTNAME` на публичный IPv4 нового VPS, затем продолжите настройку Telegram webhook в recovery-скрипте.

Telegram может сохранять старый IP в собственном DNS-кэше. Поэтому скрипт:

1. до запуска n8n переводит все первоначально опубликованные workflow в состояние `Unpublished` и отдельно отмечает содержащие Telegram Trigger;
2. после `ACTIVATE` показывает изолированные Telegram Trigger;
3. предлагает подтвердить публичный IPv4 нового VPS;
4. проверяет, что A-запись `N8N_HOSTNAME` указывает на этот IPv4 и что HTTPS health check проходит;
5. просит точно ввести `TELEGRAM`;
6. только после подтверждения возвращает изолированным workflow состояние `active=true` и перезапускает основной n8n;
7. повторно регистрирует webhook с новым origin, `ip_address`, правильным `secret_token` и `allowed_updates`;
8. проверяет URL и IP через Telegram API;
9. удаляет временный экспорт расшифрованных credentials.

Вводите:

```text
TELEGRAM
```

только после остановки старого VPS и переключения DNS.

Если `TELEGRAM` не введён, DNS/HTTPS не готовы либо credentials невозможно безопасно экспортировать, Telegram-workflow остаются неактивными. Существующий webhook старого production-сервера при этом не изменяется.

После успешной операции отправьте тестовую команду боту. Если бот не отвечает, не переопубликовывайте workflow многократно: сначала проверьте, на какой IP направлен webhook. Симптом `Connection refused` при правильном URL обычно означает, что Telegram webhook закреплён за выключенным или неверным VPS.

## 8. Обязательные проверки после восстановления

### 8.1. Контейнеры

```bash
docker ps -a --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}'
```

Основной `n8n`, workers, task runner, PostgreSQL и Redis должны быть запущены и, где предусмотрено, иметь статус `healthy`. Контейнер `n8n-import` может иметь `Exited (0)` — это нормальное успешное завершение одноразового импорта.

### 8.2. Журналы

```bash
docker logs --tail 100 n8n
docker logs --tail 100 playwright
docker logs --tail 100 caddy
```

### 8.3. Веб-интерфейс

Откройте:

- welcome page;
- выбранный при восстановлении URL n8n;
- Portainer;
- только те дополнительные сервисы, которые входят в `COMPOSE_PROFILES`.

Проверьте активные профили без вывода всего `.env`:

```bash
grep '^COMPOSE_PROFILES=' /root/selfhost-ai/.env
```

Если обычное окно браузера показывает старую страницу, а приватное — актуальную, очистите для домена cookies, local storage, cache storage и service worker, затем выполните `Ctrl+F5`.

### 8.4. n8n

Проверьте:

- вход существующим пользователем;
- наличие workflows и их опубликованное состояние;
- расшифровку credentials;
- несколько критичных workflows;
- очереди и workers;
- Playwright MCP;
- пользовательские ноды и пользовательский образ n8n;
- Telegram-бота.

### 8.5. Backup-таймер

```bash
systemctl list-timers n8n-daily-backup.timer --all
```

На боевой машине ожидается запуск в `03:30 MSK`. Дополнительно проверьте часовой пояс:

```bash
timedatectl show -p Timezone --value
```

## 9. Создание и проверка первого бэкапа после восстановления

После функциональной проверки вручную запустите штатную службу:

```bash
systemctl start n8n-daily-backup.service
```

После возврата приглашения терминала проверьте результат:

```bash
systemctl show n8n-daily-backup.service \
  -p Result -p ExecMainStatus -p InactiveEnterTimestamp

restic -r sftp:homenas-backup:/n8n-backups/restic-n8n \
  --password-file /root/.config/n8n-backup/restic-password \
  snapshots --latest 1
```

Ожидаются `Result=success`, `ExecMainStatus=0` и новый снимок `n8n-daily`. Штатный backup-скрипт в конце выполняет `restic check`; при ошибке служба не завершится успешно.

Только после этого восстановление можно считать завершённым.

## 10. Типовые проблемы

### Synology отвечает на `tailscale ping`, но SFTP/Restic выдаёт timeout

Пример:

```text
ssh: connect to host 100.x.x.x port 22: Connection timed out
Fatal: unable to open repository ...
```

Проверьте с нового VPS:

```bash
tailscale ping homenas
nc -vz -w 10 homenas 22
```

Если ping проходит, а порт 22 недоступен, проверьте:

- временный сетевой сбой и повторите проверку через несколько минут;
- DSM Auto Block и список заблокированных IP;
- брандмауэр DSM;
- правила Tailscale Access Controls для новой машины;
- включён ли SSH/SFTP на Synology.

Если сбой произошёл до фактического восстановления данных и `/root/selfhost-ai` не появился, после устранения связи скрипт можно запустить заново.

### n8n доступен локально, но не открывается в браузере

Сначала проверьте DNS и публичный IP:

```bash
printf 'VPS IP: '; curl -4 -sS https://api.ipify.org
N8N_HOSTNAME=$(sed -n 's/^N8N_HOSTNAME=//p' /root/selfhost-ai/.env)
getent ahostsv4 "$N8N_HOSTNAME" | head -n 1
```

Затем выполните локальный запрос через Caddy, приведённый выше. Если он возвращает `200`, проверьте доступ извне с другой машины. Приватное окно браузера помогает отличить серверную проблему от старой сессии или кэша.

### Welcome page показывает лишние сервисы

Проверьте фактические профили и сгенерированные данные:

```bash
grep '^COMPOSE_PROFILES=' /root/selfhost-ai/.env
jq -r '.services | keys[]' /root/selfhost-ai/welcome/data.json
```

Если данные правильные, очистите данные сайта в браузере, включая Cache Storage и service worker, затем выполните `Ctrl+F5`.

### Telegram-бот не отвечает после переключения

Наиболее вероятно, Telegram webhook всё ещё закреплён за IP другой машины. Убедитесь, что:

- старый VPS выключен;
- DNS указывает на активный VPS;
- recovery-скрипт завершил Telegram webhook failover без ошибок;
- тестовая машина не получила webhook вместо боевой.

Не публикуйте токен бота и не выводите расшифрованные credentials в журнал.

### Время снимка отличается на три часа

Recovery-скрипт показывает снимки в `Europe/Moscow`. Снимок с суффиксом `Z` в JSON Restic записан в UTC. Проверьте, какая машина его создала: тестовая восстановленная система также может включить backup-таймер после `ACTIVATE`.

На каждой активной машине проверьте:

```bash
timedatectl show -p Timezone --value
systemctl list-timers n8n-daily-backup.timer --all
```

В боевой схеме должен работать только один backup-таймер.

## 11. Возврат на старый VPS

Возврат — это отдельное переключение, а не просто включение старой машины.

1. Полностью остановите восстановленный VPS.
2. Включите старый VPS.
3. Убедитесь, что его контейнеры исправны.
4. Верните DNS A-запись на старый публичный IP.
5. Перерегистрируйте Telegram webhook на старый IP с правильным секретом n8n.
6. Проверьте сайт, credentials, workflows и бота.

Помните: база старой машины содержит состояние на момент её выключения. Изменения, сделанные на восстановленной машине после переключения, автоматически назад не переносятся.

## 12. Обновление восстановленной системы

Не обновляйте систему сразу после восстановления. Сначала завершите функциональные проверки и создайте новый успешный бэкап.

Затем обновляйте Selfhost AI штатно:

```bash
cd /root/selfhost-ai
git status
git remote -v
make update
```

В интерактивном мастере выберите тот же набор сервисов, который использовался до обновления. После обновления снова проверьте `COMPOSE_PROFILES`, контейнеры, welcome page, n8n, пользовательские ноды и Telegram-бота.

Для реальной аварии отдельно от сервера должны быть доступны:

- ссылка на этот репозиторий и эту инструкцию;
- пароль Restic;
- пароль SFTP-пользователя Synology;
- доступ к аккаунтам Tailscale, Synology, DNS и GitHub/GHCR;
- известный SSH fingerprint Synology;
- сведения о доменах и публичных IP;
