# n8n Disaster Recovery

Интерактивный bootstrap-скрипт для восстановления self-hosted n8n, PostgreSQL и Playwright из зашифрованного репозитория Restic на Synology.

Скрипт предназначен для конкретной схемы резервного копирования:

- n8n установлен через Selfhost AI;
- база данных — PostgreSQL 17;
- резервные копии Restic хранятся на Synology;
- доступ к Synology выполняется через Tailscale и SFTP;
- в снимке находятся `/root/selfhost-ai`, дамп PostgreSQL, том n8n, конфигурация Playwright, SSH-ключи и systemd-таймер резервного копирования.

Репозиторий не содержит паролей, приватных ключей, Tailscale IP или других секретов.

Файл `n8n-daily-backup` содержит актуальную версию ежедневного backup-скрипта. Он сохраняет PostgreSQL, n8n, полный том Portainer, тома Caddy, Playwright и файлы аварийного восстановления. Если штатные профили `monitoring` или `databasus` включены, скрипт также сохраняет тома Grafana, Prometheus и Databasus; при отключённых профилях они пропускаются. Redis и Docker-образы намеренно не копируются.

## Что делает скрипт

1. Проверяет, что запущен на чистом VPS от имени `root`.
2. Проверяет Ubuntu и архитектуру amd64.
3. Устанавливает Restic и необходимые утилиты.
4. Устанавливает Docker Engine из официального репозитория Docker.
5. Устанавливает Tailscale и предлагает авторизовать новый VPS.
6. Запрашивает адрес Synology, SFTP-пользователя и путь Restic.
7. Безопасно запрашивает пароль Restic и временный SFTP-пароль.
8. Показывает отпечаток SSH-ключа Synology и требует подтверждения.
9. Проверяет репозиторий и предлагает выбрать снимок.
10. Восстанавливает Selfhost AI, PostgreSQL, том n8n и Playwright.
11. Автоматически подключает Playwright к фактической Docker-сети восстановленного n8n.
12. Перегенерирует welcome-страницу по сохранённому списку активных профилей.
13. Восстанавливает автоматическое ежедневное резервное копирование.
14. Останавливается перед запуском n8n и требует ввести `ACTIVATE`.

## Требования

- новый VPS с Ubuntu 24.04 LTS amd64;
- root-доступ;
- доступ нового VPS в интернет;
- доступ к тому же аккаунту Tailscale, что и Synology;
- сохранённый отдельно пароль Restic;
- пароль SFTP-пользователя Synology;
- возможность проверить SSH fingerprint Synology;
- при необходимости — доступ к GHCR для загрузки пользовательского образа n8n;
- старый VPS должен быть остановлен перед финальной активацией.

Не устанавливайте Selfhost AI или n8n заранее: каталог `/root/selfhost-ai` восстанавливается из резервной копии целиком.

## Запуск на новом VPS

Откройте root-сессию и выполните одной строкой:

```bash
curl -fsSL https://raw.githubusercontent.com/Kalinchevs/n8n-disaster-recovery/main/restore-n8n.sh -o /tmp/restore-n8n.sh && bash /tmp/restore-n8n.sh
```

## Выбор резервной копии

После подключения к репозиторию скрипт показывает компактную таблицу из 10 последних снимков. Каждая строка содержит номер, дату и время по Москве, полный логический объём восстановления, тег и короткий ID:

```text
  No. Date and time     Size       Tag             ID
  1)  2026-07-20 14:24 25MiB      n8n-daily       2dc4637a latest
  2)  2026-07-19 03:30 24MiB      n8n-daily       a1b2c3d4
```

В строке выбора резервной копии можно указать:

- `Enter` или `latest` — самый новый снимок;
- номер `1`–`10` — снимок из показанной таблицы;
- дату `YYYY-MM-DD` — последний снимок за выбранный день;

Поиск по дате выполняется среди всех снимков репозитория, включая более старые копии, которые не попали в таблицу из 10 строк. Если на выбранную дату снимков нет, скрипт сообщит об этом и повторит запрос, не начиная восстановление. Размеры вычисляются перед показом таблицы, поэтому этот этап может занять некоторое время.

Этот вариант не требует `chmod`, сохраняет временную копию в `/tmp` и оставляет терминал доступным для интерактивного ввода паролей и подтверждений.

Если перед запуском требуется прочитать код, сначала сохраните его:

```bash
curl -fsSL \
  https://raw.githubusercontent.com/Kalinchevs/n8n-disaster-recovery/main/restore-n8n.sh \
  -o /root/restore-n8n.sh

chmod 700 /root/restore-n8n.sh
less /root/restore-n8n.sh
bash /root/restore-n8n.sh
```

## Безопасность

- Пароль Restic сохраняется в `/root/.config/n8n-backup/restic-password` с правами `600`.
- SFTP-пароль сохраняется только во временном файле в `/run` и удаляется при завершении.
- Скрипт не публикует и не отправляет введённые значения.
- n8n не запускается без точного подтверждения `ACTIVATE`.
- Скрипт отказывается перезаписывать существующий `/root/selfhost-ai` или существующие контейнеры n8n.
- SSH fingerprint Synology показывается до подключения к репозиторию.

## Что нельзя автоматизировать полностью

- вход нового VPS в Tailscale;
- проверку SSH fingerprint Synology;
- ввод паролей;
- переключение DNS;
- подтверждение, что старый VPS выключен;
- функциональную проверку credentials и workflows после восстановления.

## После восстановления

Проверьте:

```bash
docker ps
docker logs --tail 100 n8n
docker logs --tail 100 playwright
docker logs --tail 100 caddy
systemctl list-timers n8n-daily-backup.timer --all
```

Затем откройте n8n, проверьте credentials, несколько критичных workflows и Playwright MCP. После этого запустите новый резервный снимок и только затем считайте миграцию завершённой.

## Обновление восстановленной системы

После полной проверки и нового успешного бэкапа восстановленную установку можно обновлять штатно:

```bash
cd /root/selfhost-ai
git status
git remote -v
make update
```

Повторно запускать установщик Selfhost AI не требуется.

## Ограничение ответственности

Скрипт выполняет операции аварийного восстановления и изменяет систему. Используйте его только на новом или специально подготовленном VPS. Сначала прочитайте код и убедитесь, что схема резервного копирования соответствует описанной выше.
